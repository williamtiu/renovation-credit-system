# DecoFinance 系统角色权限配置分析

## 一、角色概述

系统定义了 **4 个主要角色**，每个角色有不同的职责和权限范围：

### 1. **customer** (客户)
- **职责**: 装修项目业主，发布项目、选择承包商
- **权限重点**: 项目管理、招标管理、里程碑审批

### 2. **company_user** (公司用户)
- **职责**: 装修公司代表，投标、申请贷款
- **权限重点**: 公司管理、投标管理、贷款申请

### 3. **reviewer** (审核员)
- **职责**: 平台审核人员，审批贷款申请、处理争议
- **权限重点**: 贷款审批、争议解决、查看审计报告

### 4. **admin** (管理员)
- **职责**: 系统管理员，拥有最高权限
- **权限重点**: 所有操作的完全访问权限

## 三、权限控制机制

### 3.1 核心装饰器

#### `@login_required`
```python
def login_required(f):
    @wraps(f)
    def decorated_function(*args, **kwargs):
        if 'user_id' not in session:
            flash('Please log in to access this page.', 'warning')
            return redirect(url_for('auth.login'))
        if getattr(g, 'user', None) is not None and not g.user.is_active:
            session.pop('user_id', None)
            flash('Your account is inactive.', 'danger')
            return redirect(url_for('auth.login'))
        return f(*args, **kwargs)
    return decorated_function
```
**功能**: 
- 检查用户是否登录
- 检查账户是否激活
- 未登录重定向到登录页

#### `@role_required(*roles)`
```python
def role_required(*roles):
    def decorator(f):
        @wraps(f)
        @login_required
        def decorated_function(*args, **kwargs):
            if g.user is None or g.user.role not in roles:
                flash('You do not have permission to access this page.', 'danger')
                return redirect(url_for('main.index'))
            return f(*args, **kwargs)
        return decorated_function
    return decorator
```
**功能**:
- 内置 `@login_required`
- 检查用户角色是否在允许的角色列表中
- 无权限重定向到首页

### 3.2 业务权限函数

#### `can_manage_company(user, company)`
```python
def can_manage_company(user, company):
    if user is None:
        return False
    if user.role == 'admin':
        return True
    if user.role == 'reviewer':
        return True
    return user.company_id == company.id
```
**功能**: 检查用户是否有权管理公司
- 管理员：所有公司
- 审核员：所有公司
- 公司用户：仅自己的公司

#### `require_ownership(condition)`
```python
def require_ownership(condition):
    if not condition:
        abort(403)
```
**功能**: 强制所有权检查，失败则返回 403 错误

### 3.3 特定业务逻辑访问控制

#### 贷款访问控制 `_can_access_loan(application)`
```python
def _can_access_loan(application):
    if g.user.role in ['admin', 'reviewer']:
        return True
    if g.user.role == 'company_user':
        return g.user.company_id == application.company_id
    return False
```

#### 项目访问控制 `_require_project_access(project)`
```python
def _require_project_access(project):
    if g.user.role in ['admin', 'reviewer']:
        return
    if g.user.role == 'customer':
        require_ownership(project.customer_user_id == g.user.id)
        return
    if g.user.role == 'company_user':
        company = _current_user_company()
        require_ownership(
            project.status == 'open_for_bids' or 
            _company_has_project_access(project, company.id if company else None)
        )
        return
    require_ownership(False)
```

#### 公司投标资格 `_company_has_project_access(project, company_id)`
```python
def _company_has_project_access(project, company_id):
    if not company_id:
        return False
    # 检查公司是否提交了投标或中标
    if any(bid.company_id == company_id for bid in project.bids):
        return True
    return bool(
        project.accepted_bid_id and 
        any(bid.id == project.accepted_bid_id and bid.company_id == company_id for bid in project.bids)
    )
```

## 五、关键业务流程权限

### 5.1 贷款申请流程

```
1. company_user 提交贷款申请 (POST /loans/add)
   └─ 验证：必须是公司用户，且关联自己的公司
   
2. reviewer/admin 审核贷款 (GET/POST /loans/<id>/review)
   └─ 验证：仅审核员和管理员
   
3. reviewer/admin 批准/拒绝 (POST /loans/<id>/review)
   └─ 验证：仅审核员和管理员
   
4. reviewer/admin 发放贷款 (POST /loans/<id>/disburse)
   └─ 验证：仅审核员和管理员，且贷款必须已批准
   
5. company_user/reviewer/admin 还款 (POST /loans/<id>/repay)
   └─ 验证：公司用户（自有贷款）+ 审核员 + 管理员
```

### 5.2 项目招标流程

```
1. customer/admin 创建项目 (POST /projects/add)
   └─ 验证：客户或管理员
   
2. company_user 提交投标 (POST /projects/<id>/bids)
   └─ 验证：
      - 项目状态必须是 open_for_bids
      - 公司必须通过验证（can_bid）
      - 公司尚未有活跃投标
   
3. customer/admin 接受投标 (POST /projects/<id>/bids/<bid_id>/accept)
   └─ 验证：
      - 项目业主或管理员
      - 项目状态必须是 open_for_bids
      - 尚未接受其他投标
   
4. company_user 提交里程碑 (POST /projects/milestones/<id>/submit)
   └─ 验证：
      - 必须是中标公司
      - 项目状态必须是 contracted/in_progress
      - 里程碑状态必须是 planned
   
5. customer/admin 批准里程碑 (POST /projects/milestones/<id>/approve)
   └─ 验证：
      - 项目业主或管理员
      - 里程碑状态必须是 submitted
      - 无未解决的争议
```

### 5.3 争议处理流程

```
1. customer/company_user 发起争议 (POST /disputes/add)
   └─ 验证：
      - 客户：必须是项目业主
      - 公司用户：必须参与项目
   
2. reviewer/admin 解决争议 (POST /disputes/<id>/resolve)
   └─ 验证：仅审核员和管理员
```

## 八、权限配置最佳实践

### 8.1 权限设计原则
1. **最小权限原则**: 用户仅拥有完成工作所需的最小权限
2. **职责分离**: 贷款申请和审批由不同角色执行
3. **数据隔离**: 公司用户只能访问自己公司的数据
4. **审计追踪**: 所有关键操作都有审计日志

### 8.2 安全建议
1. **定期审查 reviewer 和 admin 账户**: 这些角色权限较大
2. **监控异常操作**: 通过审计日志发现可疑行为
3. **账户 inactive 处理**: 及时禁用离职员工账户
4. **权限验证**: 所有敏感操作都需要后端验证（不仅依赖前端）

### 8.3 常见问题排查

**问题 1**: 公司用户无法编辑公司
- 检查：`user.company_id == company.id`
- 解决：确保用户关联到正确的公司

**问题 2**: 公司用户看不到贷款
- 检查：`application.company_id == user.company_id`
- 解决：确认贷款关联到正确的公司

**问题 3**: 客户无法审批里程碑
- 检查：`project.customer_user_id == user.id`
- 解决：确认项目所有权

## 九、总结

### 9.1 角色权限概览

| 权限域 | customer | company_user | reviewer | admin |
|-------|----------|--------------|----------|-------|
| **公司管理** | ❌ | ✅ (自有) | ✅ (所有) | ✅ (所有) |
| **贷款申请** | ❌ | ✅ (自有) | ❌ | ✅ |
| **贷款审批** | ❌ |  | ✅ | ✅ |
| **项目管理** | ✅ (自有) | ✅ (参与) | ✅ (所有) | ✅ (所有) |
| **争议处理** | ✅ (发起) | ✅ (发起) | ✅ (解决) | ✅ (解决) |
| **审计日志** | ❌ | ❌ | ✅ | ✅ |
| **删除操作** | ❌ | ❌ | ❌ | ✅ |

### 9.2 关键设计亮点
1. **灵活的装饰器系统**: 可组合使用 `@login_required` 和 `@role_required`
2. **细粒度访问控制**: 基于业务逻辑的权限验证
3. **完整的审计追踪**: 所有操作可追溯
4. **多语言支持**: 权限提示支持中英文

### 9.3 扩展建议
1. **增加角色**: 可添加 `auditor` (仅查看审计日志) 或 `loan_officer` (仅处理贷款)
2. **权限分级**: admin 可分为 `super_admin` 和 `operator`
3. **数据权限**: 增加部门/区域级别的数据隔离
4. **操作审批链**: 大额贷款需要多级审批
