# DecoFinance 用户需求规格说明书

## 1. 项目概述

### 1.1 项目名称
DecoFinance：装修行业信用、信托和项目融资平台

### 1.2 业务问题
香港装修公司通常面临营运资金压力、客户付款延迟以及难以向贷款机构证明运营质量的问题。传统贷款审查无法捕捉装修特定指标，如牌照有效性、职业安全健康（OSH）准备情况、纠纷历史或里程碑交付绩效。

### 1.3 产品目标
DecoFinance 提供一个装修专用的信托和融资平台，结合：

- 公司简介和合规管理；
- 信托评分和信用报告；
- 项目投标和里程碑工作流；
- 内部托管状态跟踪和纠纷处理；
- 贷款申请审查和审计可见性。

### 1.4 主要用户
| 用户类型 | 描述 | 当前系统角色 |
|------|------|------|
| 装修公司用户 | 承包商代表，管理公司简介、投标和里程碑提交 | `company_user` |
| 客户 | 物业业主或买家，创建项目并批准里程碑 | `customer` |
| 审查员 | 内部运营或财务审查员，处理纠纷和贷款决策 | `reviewer` |
| 管理员 | 平台运营者，具有审计和控制可见性 | `admin` |

## 2. 核心用户需求

### 2.1 公司用户
- 安全注册和登录。
- 维护公司、牌照、保险、职业安全健康和 ESG 信息。
- 生成信托评分并下载信用报告。
- 申请融资，可选择项目关联。
- 在验证后对客户项目投标。
- 在签约项目上提交里程碑证据。

### 2.2 客户
- 注册并创建装修项目。
- 审查竞争投标并接受一个承包商。
- 创建里程碑并审查提交的工作。
- 批准里程碑以触发释放状态。
- 在进度或质量有争议时开启纠纷。

### 2.3 审查员
- 审查和决定贷款申请。
- 解决纠纷并监控风险敞口。
- 审查公司报告、信托评分和审计活动。

### 2.4 管理员
- 监控公司、项目、贷款和纠纷的仪表板指标。
- 审查审计日志。
- 管理敏感工作流和受限操作。

## 3. 功能需求

### 3.1 身份和访问
| ID | 需求 |
|------|------|
| FUN-ID-001 | 系统应支持用户使用用户名、电子邮件、密码和角色进行注册。 |
| FUN-ID-002 | 系统应在存储前对密码进行哈希处理。 |
| FUN-ID-003 | 系统应使用经过认证的会话访问控制。 |
| FUN-ID-004 | 系统应在受保护的路由上强制执行角色和所有权限制。 |

### 3.2 公司和信托档案
| ID | 需求 |
|------|------|
| FUN-COM-001 | 系统应存储公司法律、联系、运营、牌照、保险、职业安全健康和 ESG 字段。 |
| FUN-COM-002 | 系统应使用装修特定因素计算面向信托的信用评分。 |
| FUN-COM-003 | 系统应在公司简介中缓存信托评分和风险级别。 |
| FUN-COM-004 | 系统应公开详细的公司信用报告页面，支持 PDF 下载。 |
| FUN-COM-005 | 系统应支持公司之间的报告比较。 |

### 3.3 贷款工作流
| ID | 需求 |
|------|------|
| FUN-LOA-001 | 系统应允许认证用户提交贷款申请。 |
| FUN-LOA-002 | 贷款申请可以选择引用项目。 |
| FUN-LOA-003 | 审查员和管理员用户应能够批准或拒绝申请，并提供决策详情。 |
| FUN-LOA-004 | 已批准的申请应支持拨款和还款跟踪。 |

### 3.4 项目市场
| ID | 需求 |
|------|------|
| FUN-PRJ-001 | 客户应能够创建和查看装修项目。 |
| FUN-PRJ-002 | 符合条件的公司用户应能够向开放项目提交投标。 |
| FUN-PRJ-003 | 客户应能够为每个项目接受恰好一个投标。 |
| FUN-PRJ-004 | 项目详情页面应显示投标、里程碑、纠纷和合同摘要。 |

### 3.5 里程碑、托管状态和智能合约逻辑
| ID | 需求 |
|------|------|
| FUN-CON-001 | 系统应为每个项目创建内部智能合约协议记录。 |
| FUN-CON-002 | 投标接受应激活合约并绑定选定的承包商。 |
| FUN-CON-003 | 里程碑创建应创建计划的托管状态条目。 |
| FUN-CON-004 | 里程碑提交应将合约状态更新为待审查。 |
| FUN-CON-005 | 里程碑批准应创建已释放分类账条目并更新合约余额。 |
| FUN-CON-006 | 开启纠纷应冻结相关付款状态并将合约移入冻结状态。 |
| FUN-CON-007 | 解决纠纷应重新评估合约状态。 |

### 3.6 审计和监控
| ID | 需求 |
|------|------|
| FUN-AUD-001 | 系统应记录注册、登录、评分生成、贷款决策、项目操作和纠纷操作的审计日志。 |
| FUN-AUD-002 | 仪表板应汇总公司、项目、贷款和纠纷的组合指标。 |
| FUN-AUD-003 | 系统应为自动化和测试提供 JSON 统计数据。 |

### 3.7 演示和测试支持
| ID | 需求 |
|------|------|
| FUN-OPS-001 | 系统应为演示数据提供确定性示例种子脚本。 |
| FUN-OPS-002 | 系统应提供可选数据库重置的随机数据生成器。 |
| FUN-OPS-003 | 系统应包括自动化后端和 Selenium 前端测试。 |

## 4. 业务规则

- 只有活跃且已验证的公司才能投标。
- 只有项目所有者或管理员才能接受投标。
- 只有项目所有者或管理员才能创建或批准里程碑。
- 只有指定的公司方才能提交里程碑证据。
- 纠纷在解决前冻结付款状态。
- 没有项目关联的旧记录必须继续有效。

## 5. 非功能需求

### 5.1 安全性
- 密码通过 Werkzeug 密码实用程序作为加盐哈希存储。
- 受保护的操作需要认证会话。
- 基于角色的访问控制在路由装饰器中强制执行。
- 使用基于 ORM 的查询来降低 SQL 注入风险。

### 5.2 可靠性
- 应用程序必须保持 SQLite 友好，以支持本地演示和测试执行。
- 模式创建必须通过 `db.create_all()` 工作，无需迁移依赖。

### 5.3 可用性
- UI 必须支持桌面和移动友好布局。
- 仪表板、报告、比较和项目页面必须可供非技术评估人员在演示和审查期间使用。

## 6. 验收标准

- 用户可以按角色注册和登录。
- 公司可以被创建、编辑、评分和报告。
- 信用报告 PDF 下载功能正常。
- 客户可以创建项目并接受投标。
- 里程碑可以被创建、提交和批准。
- 纠纷可以被开启和解决。
- 智能合约状态在项目详情页面可见。
- 贷款审查、拨款和还款工作流仍然有效。
- 仪表板和 API 统计数据可用。
- 随机数据生成和自动化测试成功运行。
